## 90-Second Product Demo

```text
Here is the core Boreal flow.

We start with one natural-language request. The user asks Boreal to pressure test a startup idea, design the smallest MVP, and route the right work.

Boreal Agent handles that intake for free. It can clarify the ask, structure it, and decide what route makes sense before any payment happens.

Once the work is real, Boreal turns the ask into one tracked request. It locks the best specialist route it can support on that request thread.

Now the request is no longer just chat. It has a visible route, a visible owner, and a visible next step.

For paid work, Boreal shows the quote, the seller, and the exact request that will resume after funding. The important part is that payment is not hidden. Funding is the start condition for execution.

When the owner signs, Boreal verifies the payment authorization against a real Solana mainnet transaction.

After verification, the same request resumes. It does not rematch and it does not restart somewhere else.

Then specialists execute on that same thread. Messages, evidence, delivery, and review stay attached to the work all the way through completion.
```

## 150-Second Technical Demo

```text
This demo shows Boreal's funded request flow end to end.

We begin in Boreal chat with one real ask. The user wants a startup pressure test, a smallest-MVP scope, and help routing the right work.

At this stage, Boreal Agent is free. It handles intake, clarification, and route formation before any paid specialist starts.

Once the ask is concrete enough, Boreal opens one tracked request. This is the canonical work object. It is where status, approvals, evidence, and delivery will stay.

Next, Boreal locks the best specialist route it can support. In this example, Startup Pressure Test leads first and MVP Architect follows on the same request.

At that point, Boreal shows a funded-start boundary. The UI makes three things explicit: the quote, the seller, and the exact request that will resume after payment.

That matters because execution is the thing being bought. Paid work should not begin invisibly in the background.

When the owner signs, Boreal verifies the payment authorization against a real Solana mainnet transaction before execution starts.

After verification, the same request resumes. There is no rematch, no second workflow, and no detached checkout artifact. The thread itself moves from payment required into active execution.

Now the specialists work on the same request thread. They can post progress, evidence, artifacts, and delivery into the same surface where the ask began.

That means the owner can inspect the work, review the attached outputs, and continue the conversation without leaving the request.

Boreal can also support more structured work on that same surface. Debate and Verdict keeps a moderator, two sides, and a closer inside one request thread. Solana actions can also stay explicit, owner-approved, and recorded back into the same work surface.

So the core Boreal claim is simple: the request does not disappear. It becomes a funded work thread with visible approval, routing, execution, and delivery.
```

## Delivery Notes

- Read direct, not cinematic.
- Keep pauses short.
- Stress `one tracked request`, `funding starts execution`, and `same request resumes`.
- Keep Solana practical: payment verification, approval, and execution start.
- Keep Debate and Verdict as support proof, not the main story.
