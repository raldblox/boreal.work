# Boreal Showcase Update 60

This is the dedicated one-minute Remotion cut that functions as both a project update and a product showcase.

It is not a generic launch trailer.  It is a short, high-conviction product film that proves Boreal already has a real operating surface:

- request-first intake
- routing and matching
- human, agent, and provider participation
- the four-step tracking rail
- accountable delivery
- the technical stack that keeps the request alive

## Creative Thesis

Start with the message, then earn the visuals.

The cut opens on kinetic text before any UI appears, then keeps every scene in motion.  Nothing should feel like a static screenshot.  Every card, rail, node, chip, and indicator should either be entering, pulsing, updating, scoring, or handing work forward.

## Shot Plan

### 1. Hook

Purpose:
- state the category problem before showing product

Visual:
- black field
- subtle teal motion in the background
- three large lines swap focus

Notes:
- no UI yet
- premium, quiet, deliberate
- give the three intro lines more time than before; they need to read cleanly before the UI starts

### 2. Intake

Purpose:
- show that Boreal starts from a single request in chat

Visual:
- left request rail with two existing requests first
- center intake surface with the real composer layout
- typed request animates inside the input and starts already partway formed
- `Boreal Agent` tool chip stays visible
- the send button appears only after typing is complete
- the input text clears, then the sent message appears in-thread after send
- status sequence is `structuring request` then `request created`
- a new selected request card appears in the rail
- participant icons join the selected request card

Notes:
- this should feel close to the real `next-app` shell
- the sent bubble should appear only after the composer text has faded away
- the request text is the hero, not a paragraph of explanation
- the request itself should be short and decisive
- this beat now carries more screen time so the typed request, send action, and request-creation state can land cleanly

### 3. Matching

Purpose:
- make the matching engine legible in seconds

Visual:
- request workspace card on the left
- joined participant cards on the right
- Boreal is implied as the router, but the visible list focuses on the three joined market participants
- right panel uses a real workspace-state label: `Waiting for approval`
- quote and fit score are visible per participant
- each participant has its own `Approve` and `Pass` controls
- a `See more participants` control keeps the market feeling open

Notes:
- the top joined participant should feel strongest without implying only one participant can be approved
- the participant cards should stay compact and not overflow the panel
- the request card can stay shorter than the right review panel so the approvals have room to breathe
- keep this beat shorter than the hook; it should explain the routing layer quickly, not linger
- the cut must communicate that Boreal gathered real supply from the open market, not random search results
- participant names and tags should read like real contributors joining one live request
- the message should feel two-way: Boreal gathers what exists, opens the request wider, and leaves approval with the owner
- approve comes first; pass should read as the weaker red secondary action

### 4. Route Becomes Team

Purpose:
- state the collaboration thesis cleanly before showing more UI

Visual:
- centered text only
- no extra UI clutter

### 5. Collaboration Workspace

Purpose:
- prove that a request can become a coordinated working thread

Visual:
- full-width workspace
- selected request card on the left, shaped closer to the real request rail
- participant icons join the request
- right-side participant/work cards sit inside one contained panel
- the cards show parallel contribution:
  - Boreal Agent
  - ElevenLabs voice pass
  - Remotion motion pass
  - script shaping

Notes:
- avoid overflow and route-summary clutter
- keep the collaboration view focused on who joined and what each participant is doing
- keep this beat short; it should prove active collaboration, then hand off quickly to the architecture explanation

### 6. Fulfillment

Purpose:
- show that work, files, and evidence stay attached

Visual:
- full-width request thread
- header first
- `Assigned to` appears in the header with the active participant set
- tabs on the left, compact tracking on the right
- messages after that
- explicit owner request, delivered work, and owner review beats
- inline `Delivered work` message card
- attachment pills
- preview state
- final five-star review message with star icons and follow-up comment

Notes:
- the finished work should read like a native message event, not a detached output panel
- the viewer should instantly understand who asked, who delivered, and who approved
- participant naming should stay consistent across the request rail, workspace, and thread header
- delivered work should enter with a clear message animation before the owner review appears
- this beat is intentionally shorter now; it should prove threaded delivery quickly, then hand off

### 7. Architecture Summary

Purpose:
- compress the stack into one readable mental model

Visual:
- node diagram
- animated connection points
- visible return path from fulfillment back into the chat surface
- Boreal Agent sits on the return-path junction between chat and the work thread
- explicit swarm/provider visualization on the routing side
- chat-shell and swarm dots should feel alive, with supply appearing and clearing as work gets fulfilled
- short technical labels only

Architecture nodes:
- `Request intake`
- `Request surface`
- `Live request state`
- `Matching engine`
- `Supply network`
- `Commerce spine`
- `Work thread`

Supporting labels:
- `chat, search, APIs`
- `chat UI, MCP tools, A2A`
- `context, status, approvals`
- `ranked supply routes`
- `freelancers, agents, providers`
- `wallets, x402, ACP/UCP`
- `proof, delivery`

Flow order:
- `Request intake -> Request surface -> Live request state -> Matching engine -> Supply network -> Commerce spine -> Work thread`
- return path is visualized back into the chat surface instead of using a separate delivered-reply card

Notes:
- this is the technical proof scene
- the outer scene title should name the category; the inner card title should frame demand and supply
- the node labels should read like a launch video, not an internal system diagram
- it should stay visual and light, not become a systems lecture
- this beat now gets more screen time than collaboration so the system model has room to land

### 8. Close

Purpose:
- leave one strong product sentence

Visual:
- brand close on dark teal
- no UI clutter

Primary line:
- `Submit one request.`
- `Boreal finds the best path to fulfillment.`

Support line:
- `The request stays live until the work is done.`

## Reusable Shot Strategy

This cut is designed to be recyclable:

- the hook can become a social opener
- intake can become a product ad crop
- matching can become a technical short
- participants + tracking can become a workflow explainer
- architecture can become a founder update clip

Every major scene is framed around one primary card or diagram so it can be zoomed, cropped, or trimmed later without losing clarity.

## Guardrails

- keep the UI flat and minimal
- no fake screenshots
- no empty slide energy
- no dead static panels
- no long sentences on screen
- no claims that move beyond what the product can already prove
