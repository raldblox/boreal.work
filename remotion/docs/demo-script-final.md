# Boreal Final Demo Script

Use this as the final spoken demo read.

This is the product walkthrough.

Do not use this as the category pitch.

Target runtime:

- `1:20` to `1:50`

```text
Here I'm starting in Boreal with a real request.

I ask Boreal to pressure test a startup idea, define the smallest MVP, and route the right work.

Instead of leaving that ask in a chat log, Boreal turns it into a tracked request.

Now the request becomes the working object. This is where the route, status, approvals, evidence, delivery, and review all stay attached.

Next, Boreal locks the best route it can support for this job.

At this point, the owner still stays in control. They can accept the matched route, review the quote, invite someone into the team, or open the work to the market.

When a paid route is selected, Boreal makes the funding boundary explicit.

Here you can see the quote, the seller, and the exact request that will resume after payment.

Now the owner signs.

Boreal verifies that approval against a real Solana mainnet transaction before work starts.

Once that verification lands, the same request resumes.

There is no rematch, no second workflow, and no separate handoff into another tool.

From here, the specialist team works on that same thread.

Messages stay attached. Evidence stays attached. Artifacts stay attached. Delivery stays attached. Review stays attached.

That means the full loop stays in one place, from request to funding to fulfillment.

And this same surface can already support more than one execution pattern.

It can host structured multi-agent work like Debate and Verdict, and it can connect to external supply paths like AgentCash.

So the product loop is simple.

A request comes in. The route gets locked. Funding is verified. The same thread resumes. The work gets completed in place.

That is Boreal.
```

- Read like a walkthrough, not a trailer.
